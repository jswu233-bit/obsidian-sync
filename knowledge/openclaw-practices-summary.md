## 一、余温 (@gkxspace) - 5 角色协作系统

### 核心实践
**将 OpenClaw 从单助手改造成多角色协作操作系统**

### 5 角色设计
| 角色 | 职责 |
|------|------|
| 总指挥 | 判断任务类型，@对应角色 |
| 工程师 | 给出技术方案 |
| 智库 | 做审核 |
| 创作官 | 整理最终输出 |
| 总指挥 | 收口交付 |

### 技术架构
- **5 个 AI 角色** 共享一个 OpenClaw 网关
- 跑在 **Discord 和 Telegram** 上
- 每个角色是独立的 Discord Bot

### 核心优势

#### 1. 完整处理链可见
```
用户提问 → 总指挥判断 → @对应角色
                ↓
        工程师给方案
                ↓
        智库审核
                ↓
        创作官整理
                ↓
        总指挥收口
```

#### 2. 质量兜底可靠
- 多角色交叉验证
- 每个环节有专人负责
- 避免单点失误

### 社区反馈
- ✅ "非常详细的 Agent 团队搭建教程"
- ✅ "让 OpenClaw 不再是玩具"
- ⚠️ 也有质疑："OpenClaw 原生支持 Subagent，这种设计是否多余"

### 关键洞察
**并非简单的"多开几个 bot 各聊各的"**，而是：
- 角色间有明确分工
- 有完整的协作流程
- 共享上下文和记忆

## 二、Qingyue (@yulin807) - Issue Driven 系统

### 核心实践
**基于 Issue 的多 Agent 任务管理机制**

### 系统架构
```
大龙虾（主 Agent）
    ↓ 发 Issue
小灵Team（子 Agents）
    ↓ 自动分摊任务
各角色执行 → 完成 → Close Issue
```

### 关键机制

#### 1. Issue Driven 流程
- **晚上安排任务** → 给团队分配 Issue
- **自动执行** → 各 Agent 认领并执行
- **早上起来收菜** → 查看完成的 Issue

#### 2. 任务分配机制
- 大龙虾发布 Issue
- 小弟（子 Agents）自动分摊
- 每个 Agent 选自己喜欢的任务
- 大龙虾自己上手做的任务，Issue 由他自己 Close

#### 3. 算力优化
- 避免浪费算力
- 持续找出问题修复
- 让系统更完善

### 实际应用案例

#### 案例 1: 龙虾问答 QA 网站
- 重新布置问答网站
- 方便其他龙虾提交 issue
- 解答回复，高质量 close 三条 issue
- 打造类似豆瓣小组的内容空间

#### 案例 2: GitHub 项目运营
- 把 GitHub 项目作为事业来运营
- 1 个粉丝 = 1 个用户
- Issue driven 持续迭代

### 技术要点

#### 夜间任务执行技巧
**问题**: 夜间任务一个都没做？
**原因**: systemEvent 依赖于主线程（用户-AI 对话），用户一停，任务就停

**解决方案**:
```
将 systemEvent 触发改为 isolated agentTurn
→ OpenClaw 开独立 session
→ 24/7 运行，即使夜间也持续工作
```

## 五、对我们的启发

### 可以采纳的
1. **4 角色设计** - 军师、工程师、创作官、检查官
2. **通过 @ 召唤** - 在 Discord 中直接 @ 角色
3. **Issue 管理** - 用 Issue 跟踪任务进度
4. **夜间执行** - 使用 isolated agentTurn 实现 24/7

### 需要考虑的
- 角色是否过多？是否需要简化？
- 如何平衡自动执行和人工确认？
- 如何避免算力浪费？
