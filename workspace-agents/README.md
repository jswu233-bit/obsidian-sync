# 5 角色协作系统 - 使用指南

## 系统架构

```
用户
  │
  ▼
🎯 总指挥 (Commander) - @Commander
  │
  ├──────┬──────┬──────┐
  │      │      │      │
 🧠军师 🔧工程师 ✨创作官 ✅检查官
 @军师  @工程师 @创作官 @检查官
```

## 角色速查表

| 角色 | Bot | 职责 | 召唤场景 |
|------|-----|------|----------|
| 总指挥 | @Commander | 任务分配、进度协调 | 所有任务入口 |
| 军师 | @军师 | 策略分析、方案评估 | 需要决策建议时 |
| 工程师 | @工程师 | 技术实现、代码编写 | 需要写代码/配置时 |
| 创作官 | @创作官 | 内容创作、文案撰写 | 需要写文案/报告时 |
| 检查官 | @检查官 | 质量审核、合规检查 | 需要审核内容时 |

## 使用方式

### 方式 1: 通过总指挥协调
```
用户: @Commander 我要做一个自动化备份系统

Commander: 收到任务。让我召集团队：
@军师 请分析这个需求
@工程师 准备技术方案

军师: [分析结果]
工程师: [技术方案]

Commander: 整合输出，交付用户
```

### 方式 2: 直接召唤特定角色
```
用户: @工程师 帮我写个 Python 脚本

工程师: [直接提供代码]

用户: @检查官 审核一下这段代码

检查官: [审核结果]
```

### 方式 3: 多角色协作
```
用户: @Commander 我要发布一个新产品

Commander: 召集团队：
@军师 分析市场策略
@创作官 撰写发布文案
@检查官 审核合规性

[各角色并行工作]

Commander: 汇总各角色输出，形成完整方案
```

## 典型工作流程

### 技术项目
```
用户 → @Commander
  ↓
@军师: 技术选型、架构设计
  ↓
@工程师: 代码实现
  ↓
@检查官: 代码审核
  ↓
@创作官: 技术文档
  ↓
@Commander: 整合交付
```

### 内容创作
```
用户 → @Commander
  ↓
@军师: 内容策略
  ↓
@创作官: 内容创作
  ↓
@检查官: 内容审核
  ↓
@Commander: 整合交付
```

### 复杂项目
```
用户 → @Commander
  ↓
@军师: 项目规划
  ├─ 并行 ─┬─ @工程师: 技术实现
  │        ├─ @创作官: 内容准备
  │        └─ @检查官: 持续审核
  ↓
@Commander: 整合交付
```

## 配置说明

### 每个角色的独立配置
每个角色都有自己的：
- `IDENTITY.md` - 身份定义
- `SOUL.md` - 行为准则
- `MEMORY.md` - 记忆存储
- `USER.md` - 用户信息
- `workspace/` - 工作目录

### Discord Bot 配置
每个角色需要：
1. 独立的 Discord Bot Token
2. 独立的 Bot 名称和头像
3. 连接到同一个 OpenClaw Gateway

### 共享配置
所有角色共享：
- OpenClaw Gateway
- 系统级 Skills
- 全局配置

## 最佳实践

1. **明确分工** - 每个角色专注于自己的领域
2. **避免越权** - 角色不做其他角色的工作
3. **及时沟通** - 通过 @ 及时召唤需要的角色
4. **质量把关** - 重要交付物必须经过检查官审核
5. **持续学习** - 每个角色从交互中积累经验

## 注意事项

- 总指挥是用户的主要接口
- 复杂任务建议先让军师规划
- 技术任务直接 @工程师
- 内容任务直接 @创作官
- 所有交付物建议 @检查官 审核
